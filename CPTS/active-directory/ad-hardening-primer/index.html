    Active Directory Hardening

    Цілі hardening’у

    • Ускладнити експлуатацію: Kerberoast, DCSync, ExtraSids, cross‑forest trust abuse, RBCD, unconstrained delegation.
    • Мінімізувати blast radius при компрометації одного акаунта/хоста.
    • Спрощувати виявлення аномалій та небезпечних конфігів.

    Акаунти та паролі

    • Сильні паролі + фільтри: заборонити прості/короткі/поширені; окремий пароль‑менеджер для адмінів.
    • Ротація паролів, особливо:
    • krbtgt (обидва екземпляри),
    • сервісні акаунти (SPN),
    • локальні адміни.
    • Локальний адміністратор (RID 500):
    • відключити або рандомний пароль через LAPS,
    • не використовувати один і той самий пароль на всіх хостах.
    • Tiered administration:
    • окремі акаунти для адмінки, не логінитись Domain Admin’ом на звичайні робочі станції,
    • адмінські акаунти використовувати тільки на jump‑host’ах/адмін‑станціях.
    • Protected Users:
    • додати туди чутливі акаунти (Domain/Enterprise Admins, KRBTGT, високі сервісні акаунти),
    • розуміти обмеження: нема NTLM, нема persist cache, менше lifetime квитків.

    Групи, привілеї та ACL

    • Чистка привілейованих груп:
    • мінімізувати членство в Domain Admins / Enterprise Admins,
    • ревізія Account Operators, Backup Operators, Print Operators.
    • Регулярний аналіз ACL (BloodHound / PingCastle):
    • GenericAll, GenericWrite, WriteDACL, WriteOwner на критичних об’єктах (домени, GPO, OUs, групи, високі користувачі),
    • небезпечні ACL на комп’ютерні об’єкти (RBCD, msDS-AllowedToActOnBehalfOfOtherIdentity).
    • MachineAccountQuota = 0:
    • заборонити юзерам створювати нові комп’ютерні об’єкти (складніше зловживати RBCD/noPac/ADCS‑ланцюжками).

    Сервісні акаунти та Kerberos

    • По максимуму переходити на gMSA / Managed Service Accounts замість звичайних юзерів.
    • Для сервісних акаунтів:
    • довгі випадкові паролі (20+ символів, не словник),
    • мінімально потрібні привілеї (не робити Domain Admin’ами без крайньої потреби),
    • регулярний аудит SPN‑акаунтів (Kerberoast‑цілі).
    • Вимикати/мінімізувати RC4:
    • змушувати AES (якщо всі клієнти підтримують),
    • завдяки цьому Kerberoast стає важчим або менш практичним.
    • Заборонити unconstrained delegation:
    • перевести на constrained / RBCD там, де дійсно потрібно,
    • регулярно шукати unconstrained delegation (BloodHound).

    Делегації, довіри та сегментація

    • Trust’и між доменами / лісами:
    • документувати всі trust’и,
    • по можливості не робити вечні двосторонні trust’и,
    • понижувати довіру (selective auth, SID‑filtering).
    • Cross‑forest:
    • контролювати, яким акаунтам дозволено Kerberoast’ити цільовий ліс,
    • розуміти, хто має права кшталт INBOUND/OUTBOUND trust, SIDHistory‑ланцюжки.
    • RDP / адмін‑доступ до DC:
    • тільки з певних сегментів/адмін‑хостів,
    • блокувати прямий RDP з користувацьких VLAN.
    • Вимкнути/обмежити:
    • Print Spooler на DC та чутливих серверах,
    • необов’язкові служби, які дають поверхню для relay.

    Аутентифікація, протоколи та логування

    • NTLM:
    • мінімізувати, по можливості вимикати,
    • включити NTLM auditing, шукати legacy/unknown клієнтів.
    • SMB:
    • ввімкнути SMB signing (обов’язково для DC),
    • закрутити гостьовий/анонімний доступ, RestrictNullSessAccess = 1.
    • LDAP:
    • LDAP signing, LDAPS для чутливих операцій.
    • Логування:
    • аудити входів (4624/4625, 4768–4771),
    • відстеження Kerberoast‑підозрілих шаблонів (масові AS‑REQ/TGS‑REQ),
    • моніторинг змін членства в привілейованих групах, змін ACL, створення/видалення trust’ів.

    Процеси, аудит та інструменти

    • Регулярні аудити AD:
    • BloodHound / PingCastle / Grouper,
    • фіксувати знайдені path’и до Domain/Enterprise Admin і планувати їх усунення.
    • Політика онбордингу/деонбордингу:
    • нові машини – тільки із hardened образу,
    • видаляти/деактивувати старі акаунти й комп’ютери (no stale objects).
    • Регулярні резервні копії:
    • system state DC,
    • тести відновлення (disaster recovery план).
    • Навчання адмінів:
    • не запускати браузер/пошту під Domain Admin,
    • не вводити адмінські креденшіали в неперевірене ПО,
    • базова обізнаність щодо phishing / malware / LOLBins.
