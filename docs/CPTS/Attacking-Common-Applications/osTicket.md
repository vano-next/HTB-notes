# Attacking Common Applications — osTicket

Модуль: [Attacking Common Applications](https://academy.hackthebox.com/app/module/113)
Урок: [osTicket](https://academy.hackthebox.com/app/module/113/section/1214)
Ціль: `10.129.201.88`

Відповідь: `Inlane_welcome!`

## Концепція

osTicket — це відкрита система тікетів для служби підтримки, і атака тут будується не на технічній вразливості CVE, а на людському факторі (CWE-200, information disclosure через персонал) — знайдені раніше в модулі злиті credentials дають доступ до панелі персоналу (Staff Control Panel), де в переписці з клієнтом агент підтримки порушує базові правила безпеки, надсилаючи новий пароль прямо в тексті тікета замість дзвінка.

## Крок 1 — Directory brute-force на IP (термінал)

```bash
gobuster dir -u http://10.129.201.88/ -w /usr/share/wordlists/dirb/common.txt -x php,html -t 40
```

Результат показав лише статичні маркетингові сторінки (`about.html`, `contact.html`, `services.html`) — жодних слідів osTicket на кореневому домені за IP.

## Крок 2 — Реєстрація vhost (термінал)

```bash
echo '10.129.201.88 support.inlanefreight.local inlanefreight.local' | sudo tee -a /etc/hosts
```

Пояснення: без правильного vhost-заголовка `Host:` веб-сервер (Apache/IIS з virtual hosting) віддає дефолтний сайт замість реального застосунку osTicket — це типова конфігурація shared-хостингу, де кілька сайтів сидять на одному IP.

## Крок 3 — Directory brute-force на vhost (термінал)

```bash
gobuster dir -u http://support.inlanefreight.local/ -w /usr/share/wordlists/dirb/common.txt -x php,html -t 40
```

Ключові знахідки:
- `/login.php` (422) — клієнтський портал
- `/tickets.php` (422) — список тікетів клієнта
- `/setup` (301) — інсталятор osTicket
- Відсутній прямий `/scp` у стандартному словнику `common.txt`, тому знадобилась додаткова перевірка

## Крок 4 — Перевірка типових шляхів адмін-панелі (термінал)

```bash
for path in support helpdesk osticket scp; do curl -s -o /dev/null -w "%{http_code} /$path/\n" http://10.129.201.88/$path/; done
```

Результат — усі 404, крім `scp`, який перевірили окремо на vhost:

```bash
curl -s -o /dev/null -w "%{http_code}\n" http://support.inlanefreight.local/scp/login.php
```

Результат: `422` (замість 404) — підтверджує, що `/scp/login.php` (Staff Control Panel — панель для персоналу підтримки, окрема від клієнтського порталу) існує.

## Крок 5 — Підтвердження сесії osTicket (термінал)

```bash
curl -s -D - -o /dev/null http://support.inlanefreight.local/scp/login.php | grep -i OSTSESSID
```

Результат:
```
Set-Cookie: OSTSESSID=r1k2rt0tph4o72hlao1t8iaf0v; expires=Fri, 17-Jul-2026 10:48:01 GMT; Max-Age=86400; path=/; domain=support.inlanefreight.local; HttpOnly
```

Пояснення: кука `OSTSESSID` — унікальний session identifier саме для osTicket (не generic PHPSESSID), що остаточно підтверджує технологію застосунку та готовність приймати автентифікацію.

## Крок 6 — Логін у панель персоналу через OSINT-креди (браузер)

Відкрити: `http://support.inlanefreight.local/scp/login.php`

Використати креди, знайдені раніше в модулі через OSINT/breach-дані (email `kevin@inlanefreight.local` + пароль з витоку). Логін-форма приймає як username, так і email — якщо звичайний username не проходить, спробувати email-варіант того ж акаунта.

Пояснення (чому не brute-force) — дивись розділ вище: CAPTCHA на `/captcha.php`, rate-limiting, і сама методологія уроку орієнтована на OSINT, а не перебір.

## Крок 7 — Перегляд закритих тікетів агента (браузер)

Після успішного логіну перевірити список тікетів, зокрема **Closed** — оскільки відкритих тікетів у неактивного/рідко використовуваного агента може не бути.

Знайти тікет, де клієнт — **Charles Smithson**, з розмовою про проблему з доступом/паролем.

## Крок 8 — Читання відповіді агента (браузер)

У треді тікета клієнт Charles Smithson просить допомогти з відновленням доступу і, з міркувань безпеки, просить передати новий пароль по телефону, а не письмово. Агент підтримки ігнорує цю прохання і надсилає новий пароль прямо в тексті відповіді на тікет:

```
Inlane_welcome!
```

**Відповідь: `Inlane_welcome!`**

Пояснення: пароль виглядає як типовий "стандартний пароль для нових співробітників" (`Inlane` — скорочення від назви компанії Inlanefreight, `_welcome!` — типова фраза onboarding-паролів), що робить його потенційно робочим і для інших акаунтів у компанії через password reuse.

## Чому це працює (root cause)

- **Опис:** Порушення політики безпеки з боку персоналу підтримки (CWE-200, sensitive information disclosure) — агент передав пароль у письмовій формі всупереч явному прохання клієнта та здоровому глуздом безпечної практики.
- **Технічні деталі:** Це не технічна вразливість осTicket, а процедурна проблема — система тікетів зберігає повну історію переписки в базі даних, доступну будь-кому з валідними credentials до Staff Control Panel, включно з чутливими даними, які ніколи не мали б там опинитись.
- **Ризик/Імпакт:** Отриманий пароль потенційно використовується повторно на інших сервісах компанії (Active Directory, VPN, email) через звичку співробітників використовувати однакові/подібні паролі, що відкриває шлях до подальшого lateral movement у мережі.
- **Рекомендації:** Навчання персоналу підтримки протоколам передачі чутливих даних (лише через голосовий канал з верифікацією ідентичності), впровадження DLP-рішень для сканування вихідних повідомлень на паролі/токени, регулярний аудит історії тікетів на предмет витоку credentials, примусова зміна пароля при першому вході (force password change on first login).

## Пояснення чому не брутфорсимо логін/пароль

Причина, чому в цьому уроці НЕ використовується brute-force атака на `/scp/login.php`, пов'язана з кількома технічними та методологічними факторами, які видно прямо з gobuster-виводу:

- **Captcha-захист:** Gobuster виявив ендпоінт `/captcha.php` (200 OK) — це означає, що форма логіну генерує CAPTCHA-токен, який потрібно розв'язувати при кожній спробі входу, що робить автоматизований brute-force (Hydra, Burp Intruder) непрактичним без окремого OCR/bypass-модуля.
- **Rate limiting та lockout:** osTicket за замовчуванням блокує IP або акаунт після кількох невдалих спроб логіну (типова захисна механіка), тому словниковий перебір швидко призведе до тимчасового бану, а не до знаходження пароля.
- **Методологія уроку — OSINT, а не brute-force:** Officially цей розділ Academy навчає техніці пошуку витоку credentials через сервіси типу Dehashed/breach-бази для email `kevin@inlanefreight.local`, знайденого раніше в модулі (з `/kb`, `/about.html` чи інших публічних сторінок) — це реалістичніший вектор для реального пентесту, ніж грубий перебір, бо копіює те, як справжні атакуючі використовують злиті паролі з інших сервісів (password reuse).
- **422-код на login.php:** Статус `422 Unprocessable Entity` на GET-запиті означає, що сервер очікує POST з CSRF-токеном та капчею — це додатковий бар'єр, який ще більше ускладнює автоматизований перебір без повноцінної сесії з валідним токеном.

Тобто підхід уроку — знайти вже скомпрометовані (злиті в мережі) креди через OSINT, а не атакувати саму форму логіну напряму.
