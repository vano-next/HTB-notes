Attacking Common Applications — Attacking LDAP
==============================================

Модуль: [Attacking Common Applications](https://academy.hackthebox.com/app/module/113)
Урок: [LDAP](https://academy.hackthebox.com/app/module/113/section/2153)
Ціль: `10.129.205.18`
Фінальна відповідь: `w3.css`

Ідея атаки
----------

На хості `10.129.205.18`:
- веб‑додаток на Apache 2.4.41 слухає порт `80` і показує сторінку Login;
- на порту `389` працює `OpenLDAP 2.2.X - 2.3.X`;
- логін‑форма майже напевно автентифікує через LDAP;
- якщо ввести `*` як username і password, можна обійти автентифікацію (LDAP injection / auth bypass);
- після входу на сторінку `upload.php` видно рядок `Powered by w3.css`.

Крок 1 — Nmap: виявлення HTTP + LDAP
------------------------------------

На атакуючій машині:

```bash
nmap -p- -sC -sV --open --min-rate=1000 10.129.205.18
```

З результату нас цікавить:

- `80/tcp open http Apache httpd 2.4.41 ((Ubuntu))`
- `389/tcp open ldap OpenLDAP 2.2.X - 2.3.X`
- `http-title: Login` на порту 80

Висновок:

- є HTTP‑логін, який, скоріш за все, використовує LDAP;
- LDAP не захищено (звичайний порт 389 без TLS у даній лабі);
- готуємося пробувати LDAP injection через форму входу.

Крок 2 — Перевіряємо веб‑логін
------------------------------

Через браузер:

1. Відкриваємо `http://10.129.205.18/`.
2. Бачимо сторінку з формою:
   - поле `username`;
   - поле `password`;
   - кнопка `Login`.

Фіксуємо:

- ніяких явних попереджень/капч;
- немає обмеження на спецсимволи у вводі;
- форма явно бʼє у LDAP бекенд.

Крок 3 — LDAP injection через wildcard (*)
------------------------------------------

Типова уразлива LDAP‑логіка автентифікації:

```text
(&(objectClass=user)(uid=<username>)(userPassword=<password>))
```

Якщо замість реальних значень підставити `*`, отримаємо щось на кшталт:

```text
(&(objectClass=user)(uid=*)(userPassword=*))
```

Це може матчити будь‑який запис, що веде до успішного входу.

Що робимо в інтерфейсі:

- `username`: `*`
- `password`: `*`
- тиснемо `Login`.

Очікування:

- запит до LDAP спрацьовує;
- застосунок сприймає нас як валідного користувача;
- відкривається внутрішня частина сайту (меню, сторінки типу `upload.php`).

Це й є **LDAP injection / auth bypass**: невалідація інпуту дозволяє змінити семантику фільтра й обійти перевірку.

Крок 4 — Знаходимо «Powered by»
-------------------------------

Після успішного логіну:

1. Переходимо за лінком завантаження файлів (у лабі: `http://10.129.205.18/upload.php`).
2. Дивимося низ сторінки (footer).
3. Бачимо текст:

```text
Powered by w3.css
```

Фіксуємо для відповіді:

- саме `w3.css`;
- це стандартний фронтенд‑CSS фреймворк, який тут просто згадується як «Powered by».

Відповідь
---------

**Q1:** After bypassing the login, what is the website "Powered by"?  
**A:** `w3.css`

Що тут було вразливим
---------------------

- Веб‑додаток некоректно валідовує поля `username` і `password` перед підстановкою в LDAP‑фільтр.
- Застосунок дозволяє використовувати wildcard `*`, який перетворює фільтр на «завжди істинний».
- Відсутні будь‑які додаткові захисти (rate limiting, 2FA, капча, обмеження на спецсимволи), тому LDAP injection призводить до повного байпасу автентифікації.
