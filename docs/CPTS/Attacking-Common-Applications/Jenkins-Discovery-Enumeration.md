Відповідь: `2.303.1`

## Концепція

Jenkins — це популярний сервер автоматизації CI/CD на Java, і найшвидший спосіб визначити його версію без авторизації — через HTTP-заголовок відповіді `X-Jenkins`, який Jenkins додає до кожного запиту за замовчуванням, ще до будь-якої автентифікації. Це аналогічно тому, як Tomcat розкриває версію через `/docs/`, лише тут інформація видається прямо в заголовках, а не в тілі сторінки.

## Крок 1 — Реєстрація vhost

```bash
echo '10.129.201.58 jenkins.inlanefreight.local' | sudo tee -a /etc/hosts
```

## Крок 2 — Fingerprint версії через HTTP-заголовки

```bash
curl -s -D - -o /dev/null http://jenkins.inlanefreight.local:8000/ | grep -i X-Jenkins
```

Пояснення параметрів: `-D -` виводить заголовки відповіді в stdout, `-o /dev/null` відкидає тіло сторінки (воно не потрібне), `grep -i X-Jenkins` шукає рядки з версією без урахування регістру.

Результат:
```
X-Jenkins: 2.303.1
X-Jenkins-Session: a53448c1
```

**Відповідь: `2.303.1`**

Пояснення механізму: `X-Jenkins` — це кастомний заголовок, який Jenkins Core додає до всіх HTTP-відповідей (навіть на анонімні запити до `/`), щоб інші інструменти (плагіни, CLI, API-клієнти) могли перевірити сумісність версій. `X-Jenkins-Session` — це хеш поточного запуску сервера (session ID екземпляра), який змінюється при кожному перезапуску Jenkins, і не пов'язаний з номером версії.

## Крок 3 — Альтернативний спосіб перевірки (веб-інтерфейс)

Після логіну `admin:admin` номер версії також видно в нижньому правому куті будь-якої сторінки Jenkins дрібним текстом (як згадано в підказці уроку) — це та ж сама версія `2.303.1`, просто рендериться в HTML-футері замість заголовка. Обидва методи дають однаковий результат, але заголовок `X-Jenkins` зручніший, бо не вимагає авторизації чи рендерингу сторінки.

## Крок 4 — Огляд структури Jenkins після логіну

Після авторизації варто оглянути ключові розділи:

- **Manage Jenkins** — головна панель конфігурації сервера, плагінів, користувачів та безпеки
- **New Item** — створення Job/Pipeline, звідки часто досягається RCE через Groovy Script Console
- **Credentials** — сховище збережених секретів (паролі, SSH-ключі, токени), критична ціль для post-exploitation
- **Script Console** (`/script`) — вбудована консоль виконання Groovy-коду з правами процесу Jenkins, найпряміший шлях до RCE

## Чому це важливо для подальшої атаки

Версія `2.303.1` дозволяє швидко перевірити, чи вразливий цей конкретний білд до відомих CVE (наприклад, проблем з автентифікацією чи обходом ACL у певних релізах Jenkins), а слабкі креди `admin:admin` — це класичний CWE-521 (weak credentials), який у наступних уроках модуля зазвичай веде до RCE через Script Console або через створення Pipeline-job з довільним shell-кроком.
