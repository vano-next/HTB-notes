# Attacking Common Applications — Joomla: Directory Traversal (CVE-2019-10945)

Модуль: [Attacking Common Applications](https://academy.hackthebox.com/app/module/113)

Відповідь: `j00mla_c0re_d1rtrav3rsal!`

## Концепція

CVE-2019-10945 — вразливість **Directory Traversal & Authenticated Arbitrary File Deletion** у Joomla Core версій 1.5.0–3.9.4. Уразливий компонент — файловий менеджер медіа-бібліотеки в адмін-панелі (`com_media`), який не санітизує параметр шляху при перегляді директорій, дозволяючи автентифікованому користувачу (навіть з обмеженими правами) виходити за межі папки `/images` і бачити вміст будь-якої директорії на сервері, включно з веброутом.


## Крок 2 — Спроба клонування і встановлення залежностей

```bash
git clone https://github.com/dpgg101/CVE-2019-10945.git
cd CVE-2019-10945
cat requirements.txt
pip3 install -r requirements.txt
```

Пояснення: спроба виявилась хибним стартом — цей конкретний форк (`dpgg101`) не містить файлу `requirements.txt` взагалі (`No such file or directory`), тобто скрипт не має зовнішніх залежностей окрім стандартних бібліотек Python — крок можна пропускати.

## Крок 3 — Помилка при першому запуску (bash з'їв фігурні дужки)

```bash
python3 joomla_dir_trav.py --url http://dev.inlanefreight.local --username admin --password admin --dir /flag_<hash>.txt
```

Пояснення: помилка `bash: hash: No such file or directory` виникла, бо `<hash>` — це не placeholder для копіювання буквально, а символ `<` в bash трактується як redirect input, і команда фактично намагалась виконати неіснуючий файл `hash`. Крім того, назва файлу скрипта виявилась іншою — не `joomla_dir_trav.py`, а `CVE-2019-10945.py` (перевірено через `ls -la`).

## Крок 4 — Перевірка реальної структури репозиторію

```bash
ls -la ~/CVE-2019-10945/
```

Пояснення: репозиторій містить лише один Python-файл — `CVE-2019-10945.py`, без README-прикладів запуску, тому синтаксис аргументів (`--url=`, `--username=`, `--dir=`) потрібно вгадувати зі знаком `=` замість пробілу — це нетипово для argparse (зазвичай приймає обидва варіанти, але тут форк написаний саме так).

## Крок 5 — Успішний запуск (листинг кореневої директорії)

```bash
python3 CVE-2019-10945.py --url=http://dev.inlanefreight.local/administrator --username=admin --password=admin --dir=/
```

Пояснення: параметри `--url` вказує на `/administrator` (панель, де живе вразливий медіа-менеджер), `--username`/`--password` — це **валідні креденшли адміністратора** (у цьому уроці `admin:admin` спрацював як дефолтний пароль, підібраний раніше у "Discovery & Enumeration"), а `--dir=/` — traversal-payload, який просить показати корінь файлової системи веб-сервера замість дозволеної `/images/`.

Скрипт автентифікується в адмінку, знаходить CSRF-токен, звертається до `com_media`-ендпоінту з маніпульованим параметром шляху (типово `../../../../` перед цільовою директорією) і повертає JSON-листинг файлів та папок. У виводі одразу видно нетипову для чистого Joomla-інсталу назву файлу:

```
flag_6470e394cbf6dab6a91682cc8585059b.txt
```

## Крок 6 — Читання прапора напряму через HTTP

```bash
curl -s http://dev.inlanefreight.local/flag_6470e394cbf6dab6a91682cc8585059b.txt
```

Пояснення: оскільки traversal тільки **розкриває наявність і назву** файлу (listing), а не його вміст, і файл лежить прямо у веброуті (`/var/www/dev.inlanefreight.local/`), найпростіший спосіб прочитати його — просто зробити прямий GET-запит за знайденим ім'ям, без потреби ще раз використовувати expoit. Результат:


## Чому це працює (root cause)

- **Опис:** Компонент медіа-менеджера Joomla (`com_media`) в адмін-панелі не валідує та не санітизує вхідний параметр шляху директорії при побудові файлової системи для перегляду медіа, дозволяючи послідовності `../` виходити за межі призначеної кореневої папки `images/`.
- **Технічні деталі:** Параметр `path`/`folder` конкатенується напряму до базового шляху без канонізації (`realpath()`) чи перевірки на входження в дозволену директорію (CWE-22, Path Traversal), тому запит з `--dir=/` фактично транслюється у файлову операцію `scandir()` на кореневому рівні сервера. Вразливість вимагає автентифікації (backend user з доступом до Media Manager), але не потребує високих прав — навіть Author/Editor-рівня може вистачити.
- **Ризик/Імпакт:** Дозволяє розкрити структуру та імена файлів по всій файловій системі (конфігурації, бекапи, ключі, чутливі дані), а другий компонент цього ж CVE (Arbitrary File Deletion) дозволяє видаляти довільні файли з правами веб-сервера — потенційно DoS або обхід захисних механізмів шляхом видалення `.htaccess`/security-плагінів.
- **Рекомендації:** Оновити Joomla до версії 3.9.5+ (де вразливість виправлена офіційним патчем), обмежити права backend-користувачів за принципом найменших привілеїв, додати WAF-правила для виявлення `../` послідовностей у запитах до `/administrator/index.php?option=com_media`.
