Модуль: [Attacking Common Applications](https://academy.hackthebox.com/app/module/113)

Ціль: `10.129.201.88` (gitlab.inlanefreight.local:8081)

Q1 (версія GitLab): `13.10.2`
Q2 (PostgreSQL пароль): `postgres`

## Концепція

GitLab — це веб-платформа для хостингу Git-репозиторіїв з вбудованим issue tracking та CI/CD, і в реальних пентестах вона часто стає джерелом чутливих даних не через технічну вразливість, а через випадково закомітовані секрети (паролі, API-ключі, SSH-приватні ключі) у публічних чи внутрішніх репозиторіях. Ключова особливість цього уроку — сторінка `/help` показує реальний номер версії GitLab лише авторизованим користувачам, тому якщо реєстрація відкрита, найпростіший шлях — самостійно створити акаунт замість пошуку exploit'ів для fingerprinting.

## Крок 1 — Реєстрація акаунта (браузер)

Відкрити: `http://gitlab.inlanefreight.local:8081/users/sign_up`

Заповнити форму довільними даними (наприклад `hacker:Welcome123:123@123.com`) і натиснути **Register**.

Пояснення: сторінка реєстрації доступна навіть якщо чекбокс "Sign-up enabled" вимкнено в налаштуваннях адміністратора — у такому разі форма все ще рендериться, але фактична реєстрація буде відхилена сервером; у цьому випадку реєстрація пройшла успішно, отже налаштування дозволяють вільний sign-up без підтвердження email чи модерації адміном.

## Крок 2 — Fingerprint версії через /help (браузер, після логіну)

Одразу після реєстрації система автоматично логінить нового користувача і перенаправляє на дашборд проєктів. Далі перейти на:

```
http://gitlab.inlanefreight.local:8081/help
```

Результат:
```
GitLab Community Edition 13.10.2
```

**Q1 = `13.10.2`**

Пояснення: сторінка `/help` доступна лише авторизованим користувачам і показує точний build-номер продукту — це навмисно прихована інформація для неавторизованих відвідувачів (на відміну від Jenkins чи PRTG, де версія видима одразу), тому реєстрація акаунта тут є ключовим кроком розвідки, а не опціональним.

## Крок 3 — Пошук публічних/прикладових проєктів (браузер)

Перейти на: `http://gitlab.inlanefreight.local:8081/explore`

Знайти проєкт **Inlanefreight dev** — на перший погляд виглядає як приклад/тестовий проєкт, але варто перевірити його вміст детальніше, оскільки навіть демо-репозиторії часто містять залишені тестові креди, забуті розробниками.

## Крок 4 — Перегляд файлу конфігурації тестів (браузер)

Перейти на:
```
http://gitlab.inlanefreight.local:8081/root/inlanefreight-dev/-/blob/master/phpunit_pgsql.xml
```

Вміст файлу `phpunit_pgsql.xml`:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<phpunit bootstrap="Tests/bootstrap.php" colors="true">
    <php>
        <var name="db_dsn" value="pgsql:dbname=hello_world_test;host=postgres"/>
        <var name="db_username" value="postgres"/>
        <var name="db_password" value="postgres"/>
    </php>
    ...
</phpunit>
```

**Q2 = `postgres`**

Пояснення: `phpunit_pgsql.xml` — це конфігураційний файл для PHPUnit-тестів, який визначає параметри підключення до тестової бази даних PostgreSQL (`db_dsn`, `db_username`, `db_password`); розробники часто закомітовують такі файли з дефолтними або реальними кредами "тимчасово для тестування" і забувають видалити перед пушем у репозиторій — класичний приклад CWE-798 (Use of Hard-coded Credentials).

## Крок 5 — Додаткова розвідка (опціонально, браузер)

Після логіну варто повторно перевірити `/explore` — тепер, коли акаунт авторизований, можуть з'явитись **internal**-проєкти, недоступні анонімним відвідувачам (наприклад, проєкт **Inlanefreight website** — статичний сайт компанії). Також можна використати вбудований пошук GitLab (`/search`) для швидкого пошуку ключових слів типу `password`, `secret`, `api_key` по всіх доступних репозиторіях.

## Бонус — Технікa enumeration користувачів через форму реєстрації

Навіть якщо sign-up вимкнено в налаштуваннях, сторінка `/users/sign_up` все ще дозволяє enumeration:
- Спроба зареєструвати вже зайняте ім'я користувача (наприклад `root`) поверне помилку про зайняте ім'я
- Спроба зареєструвати вже зайнятий email поверне: `1 error prohibited this user from being saved: Email has already been taken`

Це дозволяє скласти список валідних username/email для подальшого password-spraying чи credential stuffing з витоків (Dehashed), навіть без можливості завершити реєстрацію.

## Чому це важливо (root cause)

- **Опис:** Відкрита реєстрація на GitLab-інстансі (CWE-284, Improper Access Control) у поєднанні з захаращеними тестовими файлами в репозиторії дозволила будь-якому анонімному користувачу отримати чутливі дані бази даних без жодного технічного exploit'а.
- **Технічні деталі:** `/help` розкриває точну версію продукту лише авторизованим користувачам, а вільна реєстрація тривіально обходить цей бар'єр; далі закомітований `phpunit_pgsql.xml` з кредами `postgres:postgres` — це залишковий артефакт розробки, який ніхто не видалив перед публікацією репозиторію.
- **Ризик/Імпакт:** Знайдені креди PostgreSQL можуть бути дійсними не лише для тестової БД (`hello_world_test`), а й для production через password reuse; сама можливість вільної реєстрації відкриває доступ до internal-репозиторіїв з потенційно значно чутливішим кодом та секретами.
- **Рекомендації:** Вимкнути вільну реєстрацію (`Sign-up enabled` в admin settings) або вимагати підтвердження через корпоративний email + модерацію адміном, впровадити pre-commit hooks / secret scanning (наприклад GitGuardian, git-secrets) для запобігання коміту credentials, ротувати всі знайдені в історії репозиторію секрети, обмежити доступ до GitLab-інстансу через VPN/allowlist IP, якщо зовнішній доступ не є критично необхідним.
