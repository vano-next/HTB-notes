Ідея завдання
-------------

У нас є доступ по SSH до користувача `htb-student` на Ubuntu 20.04.3.  
Мета — провести базову ручну енумерацію середовища Linux (користувачі, шляхи, змінні, файли), знайти «цікавий» файл із чутливою інформацією (в цьому випадку – внутрішній скрипт, що містить флаг), і прочитати його.

Крок 1 — Підключитися по SSH
----------------------------

На Pwnbox:

```bash
ssh htb-student@10.129.113.212
```

Коли запитає пароль:

```text
HTB_@cademy_stdnt!
```

Пояснення:
- `ssh` — клієнт для підключення до Linux по мережі.
- `htb-student` — імʼя користувача на цілі.
- `10.129.113.212` — IP адреса цілі в HTB-мережі.
- пароль надає HTB Academy в тексті уроку.

Після успішного входу ти потрапиш в командний рядок (shell) під користувачем `htb-student`.

Крок 2 — Зрозуміти, де ти і ким ти є
------------------------------------

Базові команди для орієнтації:

```bash
whoami
pwd
ls -la
```

Для чого:
- `whoami` показує, під яким користувачем ти зараз працюєш (має бути `htb-student`).
- `pwd` показує поточний каталог (наприклад `/home/htb-student`).
- `ls -la` показує усі файли в директорії, включно з прихованими (починаються з `.`).

На цьому хості ти бачиш, що домашня папка майже порожня, `.bash_history` посилений у `/dev/null`, тобто історію команд не видно — це частина сценарію.

Крок 3 — Базова інформація про систему
--------------------------------------

Подивитися версію ОС:

```bash
cat /etc/os-release
```

Типовий вивід для цієї лаби:

```text
NAME="Ubuntu"
VERSION="20.04.3 LTS (Focal Fossa)"
...
```

Це підтверджує, що ми на Ubuntu 20.04 — приклад із модуля.

Далі можна глянути змінну PATH (де система шукає команди):

```bash
echo $PATH
```

На цьому хості PATH вже показано в модулі, тож ми просто фіксуємо, що тут стандартний набір директорій (`/usr/bin`, `/bin`, `/usr/sbin` тощо).

Крок 4 — Подивитися інших користувачів
--------------------------------------

Хоч ми не повторюємо твої команди, логіка уроку така:
- дивимося, які домашні директорії існують;
- цікавим може бути інший юзер, наприклад `lab_adm`.

Команда:

```bash
ls /home
```

Побачиш:

```text
htb-student  lab_adm
```

Це каже, що є ще один користувач `lab_adm` з власною домашньою директорією — потенційне джерело «цікавих» файлів (конфіги, скрипти, історія команд).

Крок 5 — Зайти в /home іншого юзера (читати, не змінювати)
---------------------------------------------------------

Переходимо:

```bash
cd /home/lab_adm
ls -la
```

Пояснення:
- ми **не міняємо** власника файлів, просто читаємо те, що світиться і не потребує root;
- в Ubuntu часто домашні папки інших юзерів частково відкрити (права `drwxr-xr-x` на каталог).

Тут ти бачиш:
- `.bashrc`, `.profile`, `.viminfo` тощо;
- але нічого явно з флагом, тому йдемо далі.

Крок 6 — Шукати «цікаві» файли по всій системі
---------------------------------------------

Модуль радить:
- шукати файли на кшталт `*flag*`, але це може давати багато шуму;
- шукати рядок `HTB{` вмісту файлів, бо HTB-флаги починаються саме так.

### 6.1 Пошук файлів із назвою `*flag*`

Команда:

```bash
find / -type f -iname "*flag*" 2>/dev/null
```

Пояснення:
- `find /` — шукаємо з кореня всього файлового дерева (`/`).
- `-type f` — тільки файли.
- `-iname "*flag*"` — імʼя файлу містить слово `flag` (регістр не важливий).
- `2>/dev/null` — помилки прав доступу (Permission denied) відправляємо в «чорну діру», щоб вони не заважали.

На цій машині це повертає багато системних файлів, але **не глава з HTB-флагом** — отже, йдемо іншим шляхом.

### 6.2 Пошук по вмісту файлів `HTB{`

Команда:

```bash
grep -R -n -l 'HTB{' / 2>/dev/null
```

Пояснення:
- `grep` — пошук текстових рядків.
- `-R` — рекурсивно по директоріях.
- `-n` — показувати номер рядка (не критично, але зручно).
- `-l` — показувати лише імʼя файла, який містить збіг.
- `'HTB{'` — шаблон флагу HTB.
- `/` — стартуємо з кореня всіх файлів.
- `2>/dev/null` — сховати помилки «Permission denied».

На цій цілі команда повертає:

```text
/lib/int-check.sh
/usr/lib/int-check.sh
```

Це вже **дуже конкретно**: два скрипти `int-check.sh` у `/lib` і `/usr/lib`, де є рядок із `HTB{`.

Крок 7 — Піти до /lib і прочитати скрипт
----------------------------------------

Зосередимося на `/lib/int-check.sh` (можна й `/usr/lib/int-check.sh`, вони однакові).

Переходимо:

```bash
cd /lib
ls
```

Тут ти бачиш багато системних директорій (`systemd`, `networkd-dispatcher`, `nagios`, `snapd`, тощо), а також файл:

```text
int-check.sh
```

Переконуємося, що файл існує:

```bash
ls int-check.sh
```

Тепер читаємо його вміст:

```bash
cat int-check.sh
```

Вивід:

```text
HTB{1nt3rn4l_5cr1p7_l34k}
```

Це і є флаг, який треба здати в Academy.

Крок 8 — Відповідь для модуля
-----------------------------

У полі відповіді в секції питання вводиш:

```text
HTB{1nt3rn4l_5cr1p7_l34k}
```

Якщо будеш перевіряти через `grep` ще раз — побачиш той самий рядок у `/usr/lib/int-check.sh`.

Що тут було «цікавим» із точки зору привілей-ескалації
------------------------------------------------------

- Використання `grep -R 'HTB{' /` — це один з класичних лайфхаків для lab/CTF-середовищ, де флаги лежать у текстових файлах.
- Скрипт у `/lib` із назвою `int-check.sh` — приклад «внутрішнього» системного скрипта, який може:
  - містити чутливі дані (флаги, паролі, токени);
  - виконуватися як root (у реальних системах, через cron чи systemd).
- У реальному pentest-оточенні подібний скрипт можна було б використати:
  - як точку для **перехоплення виконання** (якщо він запускається з підвищеними правами);
  - як джерело паролів/ключів до інших сервісів.

Висновок:
- Модуль демонструє, що **ручна енумерація** (find/grep по всій системі) здатна знаходити «заховані» скрипти й секрети, навіть якщо немає очевидних вразливостей у сервісах чи конфігах.
